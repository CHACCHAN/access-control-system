# 外部 API 連携仕様

在室管理サーバーとの連携。エンドポイント URL・API トークン・リクエストボディの
形は全て設定画面から変更できる(再ビルド不要)。

## 認証

全 API 共通で `Authorization` ヘッダーに設定画面の「API トークン」を**そのまま**送る。
トークンは `Bearer xxx` の形で丸ごと入力してもらう運用(アプリ側で `Bearer ` を
重ねて付けない)。

## 1. メンバー一覧取得

- `GET {getEndpoint}`
- レスポンス: メンバー配列

```ts
interface Member {
  username: string;
  name: string;
  status: "在室" | "外出" | "帰宅";
  descriptor?: number[];  // 顔特徴ベクトル(512次元、未登録なら省略)
}
```

- 起動時(ブートチェックの「サーバー接続」)と、メイン画面表示時、
  WebSocket の更新シグナル受信時に取得する。

## 2. 在室状況更新

- `POST {attendanceEndpoint}`
- ボディはテンプレート(設定「APIボディ」)から生成。既定:

```json
{ "userName": "{{username}}", "name": "{{name}}", "newStatus": "{{status}}" }
```

- プレースホルダー `{{username}}` / `{{name}}` / `{{status}}` が実値に置換される。
  置換はノード単位(文字列・配列・オブジェクトいずれも可)。
- 成功時はローカルの一覧表示を即時更新する(再取得はしない)。

## 3. 顔特徴ベクトル登録

- `POST {postEndpoint}/{username}`
- ボディはテンプレートから生成。既定:

```json
{ "descriptor": "{{descriptor}}" }
```

- `{{descriptor}}` は number[](**512次元**・ArcFace)に置換される。
- サーバー側の格納カラム(pgvector 想定)は **`VECTOR(512)`** である必要がある。
  旧 faceapi.js 時代の `VECTOR(128)` のままだと、次元不一致で登録 API が **400** を返す
  (アプリは常に 512次元を送る)。その場合はサーバーのカラム型を `VECTOR(512)` へ
  マイグレーションする。登録失敗時はサーバーのレスポンス本文をエラーに含めて表示する。

## 4. 更新シグナル(WebSocket)

- `{wsEndpoint}` へ接続し、受信した JSON メッセージの
  `{wsSignalField}`(既定 `message`)フィールドが `{wsSignalValue}`(既定 `update`)と
  一致したら、メンバー一覧を裏側で再取得する(スケルトンは出さない)。
- 切断時は 5 秒間隔で自動再接続する。JSON でないメッセージは無視する。

## ボディテンプレートの仕様

- テンプレートは通常の JSON。値の位置に `"{{key}}"` という文字列を置くと、
  そのノード全体が実値に置き換わる(`src/shared/lib/apiBodyTemplate.ts`)。
- バックエンドのフィールド名変更には、設定画面でテンプレートを書き換えるだけで
  追従できる。不正な JSON は設定画面の検証で弾かれる。

## エラーハンドリング

- 接続レベルの失敗(DNS・TLS・タイムアウト)と HTTP エラー(4xx/5xx)を区別して
  ログ・ユーザー向けメッセージを出す。
- HTTPリクエストはレスポンス本文の受信を含め10秒でタイムアウトし、画面や起動診断を
  永久待機させない。新しい再取得が必要な場合は古い別endpointへの通信を中断する。
- メンバー一覧取得失敗時は一覧パネルにエラーメッセージを表示する。
- メンバーJSONはusername・name・status・descriptorを実行時検証し、不正データを
  描画層へ渡さない。WebSocket通知の連続受信はまとめ、取得中なら完了後に1回だけ再取得する。
